# 블로그 문체 가이드 (crowwan.github.io)

이 문서는 **블로그 글을 쓰거나 다듬는 세션**(이 레포 cwd)이 일관된 톤으로 작성하도록 돕는다.
회사 작업 내용을 일반화해 올릴 때, 소재는 볼트 `blog-handoff/`에서 넘어온다(아래 파이프라인 참고).

---

## 글의 정체성

> **"삽질과 발견의 기록."** 결론만 깔끔히 정리한 튜토리얼이 아니라, **헤맨 과정을 솔직하게 보여주고 거기서 배운 것**을 나누는 경험담형 블로그.

독자는 "나도 똑같이 헤맸는데" 하고 공감하러 온다. 그러니 **실패를 숨기지 않는다.**

---

## 톤 & 보이스

- **친근한 경어체** — "~습니다", "~했어요"를 섞어 씀. 딱딱한 "~한다"체 금지.
- **솔직함** — "솔직히 번거롭습니다", "내 탓", "뭘 하고 있는 거지?" 같은 자기고백을 살린다.
- **가벼운 농담** — 취소선(`~~(내 탓)~~`)으로 셀프 디스, 과장된 현타 표현 OK.
- **독자에게 말 걸기** — "~하지 않을까 싶어서요", "맞습니다" 같은 대화체.
- **과장 금지** — "혁신적", "완벽한" 같은 마케팅 어휘 X. 담백하게.

## 구조 패턴

1. **인용구 훅으로 시작** — 글 맨 앞에 `>` blockquote로 핵심 질문이나 던진 한마디를 둔다. frontmatter `description`도 같은 문장으로 맞춘다.
   ```markdown
   > "클로드 코드로 AI 비서 같은 걸 만들 수 있을까?"
   > 그 질문에서 시작된 삽질과 발견의 기록입니다.
   ```
2. **이모지 섹션 헤더** — `## 🤔 시작`, `## 🎯 목표`, `## 🐳 1차 시도`, `## 🤯 현타`, `## ⚠️ 늪`, `## ✅ 결론`. 감정·단계가 드러나는 이모지를 쓴다.
3. **서사 곡선** — 동기 → 목표 → 1차 시도(실패) → 우회(또 실패) → 현타 → 돌파구 → 배운 것. 직선적 성공담 X.
4. **대조 보여주기** — "원래 목표 vs 현재 상황" 같은 블록으로 헤맨 지점을 시각화.

## 자주 쓰는 장치

- **비교는 표로** — 접근법 4가지, 수정 요청 목록 등은 마크다운 표.
- **실제 명령어 + 실패 출력** — 코드블록에 진짜 입력과 `❌ ...` 에러 출력을 같이.
- **인라인 인용구로 리듬** — 중간중간 `>` 로 그때 든 생각/혼잣말을 끼워 넣는다.
- **체크/이모지 상태 표시** — `✅ 시도 → 실패`, `🔄 ...중`.

## frontmatter 스키마

`src/content.config.ts` (zod). 필수는 `title`, `pubDate`.

```yaml
---
title: "큰 제목 - 부제로 호기심 유발"
pubDate: 2026-06-09          # YYYY-MM-DD
description: "맨 앞 인용구와 같은 문장 (선택, 권장)"
tags: ["Claude", "자동화"]   # 선택
heroImage: "..."             # 선택 (원격 URL/로컬 경로)
updatedDate: 2026-06-10      # 선택
---
```

- **파일명**: 한글 제목을 kebab-case로 (`src/content/blog/제목-부제.md`). 공백은 `-`.
- `tags`는 기존 글의 태그 어휘를 재사용(`Claude`, `개발`, `자동화`, `TDD`, `MCP` 등).

## 체크리스트 (발행 전)

- [ ] 맨 앞 인용구 훅이 있는가 / `description`과 톤이 맞는가
- [ ] 실패·삽질 과정이 솔직하게 담겼는가 (성공담만 아님)
- [ ] 경어체 일관 / 마케팅 어휘 없음
- [ ] 섹션 헤더에 단계·감정 이모지
- [ ] **회사 비식별화 통과** — 내부 서비스/코드명·고객사·티켓번호·사내 URL·사번 없음 (볼트 `handbook/blog-publishing-pipeline.md` 체크리스트)

## 회사 작업 → 블로그 파이프라인 (요약)

1. **작업 세션**(`~/Works/devops/*`)에서 그날 배운 것 설명 → Claude가 코드 맥락 보고 민감도 판단 → **1차 비식별화**한 소재를 볼트 `blog-handoff/`에 저장.
2. **이 블로그 세션**은 `~/work-vault/blog-handoff/`만 읽어 → 이 가이드 문체로 글 작성 → **2차 비식별화 검증** → 발행.

> 회사 코드는 이 세션에서 **절대 직접 읽지 않는다.** 소재는 항상 `blog-handoff/`를 거쳐서만 들어온다.
